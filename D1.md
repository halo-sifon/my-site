## 从零接入 D1

本次接入使用 Cloudflare 官方 Wrangler migrations、原生 D1 Binding
API 和 OpenNext `getCloudflareContext()`，不引入 ORM。

### 升级 Wrangler 和测试依赖

#### 命令格式

```bash
npm install -D wrangler@latest vitest@latest @cloudflare/vitest-pool-workers@latest
```

#### 作用

- 将项目内 Wrangler 升级到最新 4.x。
- 安装 Vitest 和 Cloudflare Workers 官方测试池。
- 让 D1 测试运行在本地 Workers runtime 和 Miniflare D1 中。

#### 实际结果

```text
wrangler: 4.104.0
vitest: 4.1.9
@cloudflare/vitest-pool-workers: 0.16.19
```

### 创建远程 D1 数据库

#### 命令格式

```bash
npx wrangler d1 create <database-name> \
  --location <location> \
  --binding <binding-name> \
  --update-config
```

#### 作用

创建远程 D1 数据库，并将数据库 binding 自动写入 `wrangler.jsonc`。

#### 参数说明

- `<database-name>`：Cloudflare 中的数据库名称。
- `--location`：数据库主位置提示。本项目使用 `apac`。
- `--binding`：代码中访问数据库的变量名。本项目使用 `DB`。
- `--update-config`：自动更新当前项目的 Wrangler 配置。

#### 实际示例

```bash
npx wrangler d1 create my-site-db \
  --location apac \
  --binding DB \
  --update-config
```

创建结果：

```text
database_name: my-site-db
database_id: b234e386-b115-4575-ab37-4e915fc91306
binding: DB
region: APAC
```

#### 风险说明

- 该命令会创建真实的远程 Cloudflare 资源。
- 数据库名称不能用于代码访问，代码应使用 binding。
- 不要手动伪造或复用其他数据库的 `database_id`。

### 配置 D1 binding

`wrangler.jsonc` 中的实际配置：

```jsonc
{
  "d1_databases": [
    {
      "binding": "DB",
      "database_name": "my-site-db",
      "database_id": "b234e386-b115-4575-ab37-4e915fc91306",
      "migrations_dir": "migrations",
    },
  ],
}
```

字段作用：

- `binding`：通过 `getCloudflareContext().env.DB` 访问数据库。
- `database_name`：Wrangler 命令使用的稳定数据库名称。
- `database_id`：Cloudflare 为远程数据库生成的唯一 ID。
- `migrations_dir`：Wrangler 查找 SQL migration 的目录。

### 生成 Cloudflare 类型

#### 命令格式

```bash
npm run cf-typegen
```

#### 作用

根据 `wrangler.jsonc` 重新生成 `cloudflare-env.d.ts`，使 TypeScript 识别
`DB: D1Database`。

#### 验证方式

```bash
rg "DB: D1Database" cloudflare-env.d.ts
```

### 创建 migration

#### 命令格式

```bash
npx wrangler d1 migrations create <database-name> <message>
```

#### 作用

在 `migrations/` 中创建带顺序编号的 SQL migration 文件。

#### 实际示例

```bash
npx wrangler d1 migrations create my-site-db create_notes_table
```

生成文件：

```text
migrations/0001_create_notes_table.sql
```

该 migration 创建 `notes` 表，包含：

- `id`
- `title`
- `content`
- `created_at`
- `updated_at`

### 应用本地 migration

#### 命令格式

```bash
npx wrangler d1 migrations apply <database-name> --local
```

#### 作用

只对 `.wrangler/state/`
中的本地 D1 应用尚未执行的 migrations，不修改远程数据库。

#### 实际示例

```bash
npx wrangler d1 migrations apply my-site-db --local
```

#### 验证方式

```bash
npx wrangler d1 migrations list my-site-db --local
```

输出 `No migrations to apply`，说明本地 migration 已全部应用。

### 应用远程 migration

#### 命令格式

```bash
npx wrangler d1 migrations apply <database-name> --remote
```

#### 作用

将尚未执行的 migrations 应用到 Cloudflare 上的远程 D1 数据库。

#### 实际示例

```bash
npx wrangler d1 migrations apply my-site-db --remote
```

#### 风险说明

- 该命令会修改真实远程数据库结构。
- 应先在本地应用并完成测试，再执行 `--remote`。
- 已执行的 migration 不应修改；后续结构变化必须创建新 migration。

#### 验证方式

```bash
npx wrangler d1 migrations list my-site-db --remote
```

查询远程表结构：

```bash
npx wrangler d1 execute my-site-db \
  --remote \
  --command "SELECT name, sql FROM sqlite_schema WHERE type = 'table' AND name = 'notes'"
```

#### 执行结果

- 本地和远程均无待应用 migration。
- 远程 `notes` 表创建成功。
- 远程查询由 APAC 区域的 HKG 节点提供。

## D1 CRUD 接口规范

每个数据模型应显式实现自己的字段校验和 SQL，不提供可操作任意表的通用接口。

`notes` 是首个完整 CRUD 模板：

| 方法     | 路径             | 作用         |
| -------- | ---------------- | ------------ |
| `GET`    | `/api/notes`     | 分页查询笔记 |
| `POST`   | `/api/notes`     | 新增笔记     |
| `GET`    | `/api/notes/:id` | 查询笔记详情 |
| `PATCH`  | `/api/notes/:id` | 部分更新笔记 |
| `DELETE` | `/api/notes/:id` | 硬删除笔记   |

代码通过以下方式获取 binding：

```ts
const db = getCloudflareContext().env.DB;
```

所有动态 SQL 参数使用 prepared statements 和 `bind()`，禁止拼接用户输入。

统一响应结构：

```json
{
  "code": 0,
  "data": {},
  "message": "success"
}
```

响应码常量：

```text
SUCCESS: 0
BAD_REQUEST: 400
NOT_FOUND: 404
INTERNAL_ERROR: 500
```

## 删除远程 D1 数据库

### 命令格式

```bash
npx wrangler d1 delete <database-name> --skip-confirmation
```

### 作用

永久删除 Cloudflare 上指定的远程 D1 数据库及其中的全部数据。

该命令只删除远程数据库，不会自动修改本地的
`wrangler.jsonc`、迁移文件、类型声明或业务代码。

### 参数说明

- `npx wrangler`：运行项目本地安装的 Wrangler CLI。
- `d1 delete`：执行删除 D1 数据库操作。
- `<database-name>`：需要删除的远程 D1 数据库名称，也可以使用 Wrangler 配置中的 D1
  binding。
- `--skip-confirmation`：跳过 Wrangler 的交互式确认。使用此参数前必须明确确认数据库名称，因为删除操作不可逆。

### 实际示例

2026-06-24 删除远程数据库 `my-site-db`：

```bash
npx wrangler d1 delete my-site-db --skip-confirmation
```

被删除的数据库信息：

```text
database_name: my-site-db
database_id: b0c2258b-4cf5-4bbc-b90c-90dec63b1924
```

### 风险说明

- 删除操作不可逆。
- 数据库中的表、数据和远程迁移记录会被永久删除。
- 删除前必须核对数据库名称和 ID。
- 如果需要保留数据，应先使用 `wrangler d1 export` 导出备份。
- 删除后，本地 `wrangler.jsonc` 中原有的 `database_id`
  将失效，需要在重新创建数据库后更新。

### 验证方式

列出当前 Cloudflare 账号中的远程 D1 数据库：

```bash
npx wrangler d1 list --json
```

检查输出中是否还存在目标数据库的名称或 ID。目标数据库不存在，说明远程删除成功。

### 执行结果

Wrangler 返回：

```text
Deleted 'my-site-db' successfully.
```

执行验证命令后，远程数据库列表中不再包含 `my-site-db`，删除成功。

## 重置 Wrangler 全部本地状态

### 命令格式

```bash
rm -rf .wrangler/state
```

### 作用

删除 Wrangler 和 Miniflare 在当前项目中生成的全部本地模拟数据，将本地 Cloudflare 开发环境重置为空状态。

Wrangler 没有提供 `wrangler d1 delete --local`
命令。本地 D1 和其他本地 Cloudflare 资源都保存在 `.wrangler/state/`
中，因此需要通过删除该目录完成重置。

该操作不会删除任何远程 Cloudflare 资源，也不会修改
`wrangler.jsonc`、迁移文件、类型声明或业务代码。

### 参数说明

- `rm`：删除文件或目录。
- `-r`：递归删除目录及其内容。
- `-f`：不进行交互式确认，并忽略不存在的目标。
- `.wrangler/state`：Wrangler 和 Miniflare 保存本地模拟数据的目录。

### 实际示例

重置当前项目的全部 Wrangler 本地状态：

```bash
rm -rf .wrangler/state
```

### 风险说明

- 删除后，本地 D1 中的表、数据和本地迁移执行记录会丢失。
- 本地 Cache、KV、R2、Durable Objects 等其他模拟数据也会一并丢失。
- 该操作不可恢复，除非提前备份本地 SQLite 文件。
- 该命令只适合需要彻底重置本地 Cloudflare 开发状态的场景。
- 再次运行 Wrangler 本地开发或本地资源命令时，所需目录会自动重新创建。
- 如果需要恢复数据库结构，需要重新执行本地 migrations。

### 验证方式

检查本地状态目录是否仍然存在：

```bash
test ! -d .wrangler/state && echo "Wrangler 本地状态已清除"
```

输出 `Wrangler 本地状态已清除`，说明本地状态目录已删除。

### 执行结果

2026-06-24 实际只删除了本地 D1 子目录：

```text
.wrangler/state/v3/d1
.wrangler/state/v3/v3/d1
```

本次没有执行
`rm -rf .wrangler/state`，因此 Cache、Images 等其他本地状态仍然保留。

## 清理项目中的 D1 实现

### 命令格式

该操作没有单一的 Wrangler 命令，需要根据项目实际结构删除 D1 相关文件，并从
`wrangler.jsonc` 中移除 `d1_databases` 配置。

删除文件或目录的通用格式：

```bash
rm -rf <d1-related-path>
```

移除配置后重新生成 Cloudflare 类型：

```bash
npm run cf-typegen
```

### 作用

清理项目中的 D1 迁移文件、API、测试页面、数据库访问封装和 Wrangler
binding，使项目恢复到未接入 D1 的状态。

该操作只修改本地项目文件，不会创建、删除或修改远程 Cloudflare 资源。

### 参数说明

- `rm -rf`：递归、强制删除指定文件或目录。
- `<d1-related-path>`：项目中确认仅由 D1 功能使用的文件或目录。
- `npm run cf-typegen`：根据当前 `wrangler.jsonc` 重新生成
  `cloudflare-env.d.ts`。
- `d1_databases`：`wrangler.jsonc` 中声明 D1 binding 的配置项。

同时移除`wrangler.jsonc` 中的 `d1_databases` 配置。

### 风险说明

- 删除迁移文件后，无法通过原迁移记录重新创建旧数据库结构。
- 删除 API 和数据库封装后，依赖这些模块的代码必须同步清理。
- 移除 binding 后，代码继续访问原 binding 会导致类型检查或运行时错误。
- 重新生成的 `cloudflare-env.d.ts` 不应再包含原 D1 binding。
- 该操作不会影响已经存在的远程数据库；远程资源需要单独使用 Wrangler 删除。

### 验证方式

检查项目中是否仍有旧 D1 binding、数据库名称或测试表引用：

然后运行项目验证：

```bash
npm run lint
npm run build
```

### 执行结果

已删除原有 D1 迁移、API、测试页面和数据库访问封装，并移除 Wrangler D1 binding。

`npm run cf-typegen` 执行成功，生成的 `cloudflare-env.d.ts` 不再包含旧 D1
binding。
